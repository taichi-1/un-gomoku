# ungomoku-ml

AlphaZero-style training pipeline for the un-gomoku CPU opponent.
PyTorch self-play training happens here; the exported ONNX model is served to
the browser from `apps/web/public/models/`.

## Setup

```bash
uv sync          # from ml/ (Windows dev machine pulls the cu132 torch wheel)
```

## Commands

Run from `ml/` (or `uv run --directory ml ...` from the repo root):

```bash
uv run ungomoku-ml train --config configs/default.yaml            # full training run
uv run ungomoku-ml train --config configs/default.yaml --resume runs/default/latest.pt
uv run ungomoku-ml train --config configs/smoke.yaml               # minutes-long sanity loop
uv run ungomoku-ml selfplay --config configs/default.yaml --games 50   # throughput check
uv run ungomoku-ml arena --config configs/default.yaml --p1 runs/default/best.pt --p2 heuristic
uv run ungomoku-ml export --ckpt runs/default/best.pt --out ../apps/web/public/models/ungomoku-v1.onnx
uv run ungomoku-ml gen-encoding-vectors                            # TS<->py feature parity fixture
uv run pytest -m "not slow"                                        # fast tests
uv run pytest -m slow                                              # ONNX export round-trip
```

Checkpoints, metrics (`metrics.jsonl`), and `best.pt` land in `runs/<run_name>/`
(gitignored). Resuming restores net/optimizer/generation but not the replay
buffer; the first resumed generation refills it.

## How it works

- **Game structure**: a move submits 1-5 candidate cells; success probability
  depends on the count (0.5..0.9) and the placed cell is uniform among the
  candidates. For a fixed size k the optimal subset is the top-k cells by
  per-cell value Q, so move selection reduces to comparing
  `EV(k) = p(k) * mean(top-k Q) + (1-p(k)) * Q_pass` for k = 1..5
  (`mcts/ev.py`, mirrored by the TS engine).
- **Net** (`net.py`): ResNet (default 6 blocks x 64 ch, ~0.9 MB of weights),
  policy over 225 cells + scalar value, to-move-relative planes (`encoding.py`).
- **Search** (`mcts/`): Gumbel-style root (top-m cell sampling + sequential
  halving) over a tree whose decision nodes have per-cell children plus a
  "pass" child for failed turns; chance is sampled during descent and values
  back up through the exact EV operator.
- **Self-play** (`driver.py`): hundreds of concurrent games advance generator-
  based searches in lockstep; all pending evals are batched into single GPU
  forwards.
- **Training** (`train_loop.py`): generational loop — selfplay -> gradient
  steps (policy CE + value MSE, value target mixes outcome with root search
  value) -> arena gate vs `best.pt` (promote at >= 55%).
- **Rule parity**: `rules/` is an exact port of `packages/core`; fixtures
  generated by `bun run gen:parity` are replayed in `tests/test_rules_parity.py`.
  Feature encoding parity for the browser engine uses
  `tests/fixtures/encoding-vectors.json`.

## Strength gates (M3)

1. score >= 0.90 vs `random`
2. score >= 0.70 vs `heuristic`
3. score >= 0.55 over 400 games vs the legacy expectiminimax (hard) via
   `tools/parity-arena` (the ship gate)
